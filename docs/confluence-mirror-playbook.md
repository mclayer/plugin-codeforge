# Confluence 읽기 사본(미러) 운영·재사용 안내서

> 이 안내서는 codeforge 본체 문서를 Confluence 에 "읽기 편한 한 방향 사본(미러)"으로 정리해 두고, 새 소비자 프로젝트에 같은 방식을 복제하며, git 문서가 바뀔 때 사본을 다시 맞추는 실무 절차를 담는다. 문서 작성·렌더 기준의 단일 원본(SSOT)은 [문서 가독성·소통 표준(설계결정 123)](doc-readability-standard.md)이며, 기준이 충돌하면 그 문서가 우선한다.

## 개요

- 모든 문서의 **정본(진실의 원천)은 git 저장소**이다. Confluence 는 그 정본을 사람이 읽기 편하게 정리한 **한 방향 사본**일 뿐이다.
- 사본은 git 에서 Confluence 로만 흐른다(한 방향). Confluence 페이지를 직접 고치는 일은 금지한다. 직접 고치면 정본과 어긋나며, 다음 사본 갱신 때 덮어쓰여 사라진다.
- 현재 codeforge 본체 문서는 예시 공간인 CFP(CodeForge) 공간에 이 방식으로 정리돼 있다. 새 소비자(예: mctrader)는 같은 체계를 자기 공간에 복제한다.

| 구분 | 위치 | 성격 |
|---|---|---|
| 정본 | git 저장소(설계결정 · 도메인 지식 · 계약 · 안내서 · 아키텍처) | 진실의 원천. 모든 편집은 여기서 |
| 사본 | Confluence 공간(위 정본을 읽기용으로 렌더) | 읽기 전용. 한 방향 복제, 직접 편집 금지 |

> 사본 대상은 **사람이 읽는 문서**로 한정한다. 작업 단위 파일·고침 기록·근거 묶음 같은 진행 산출물은 사본으로 만들지 않는다(아래 §"사본 금지 다섯 영역" 참조).

## 배경

git 저장소는 변경 이력·검토·검사가 모이는 작업의 중심이지만, 읽기 환경으로는 불편하다. 반대로 Confluence 는 읽기·검색·공유에 강하나 편집을 열어 두면 정본이 둘로 갈라진다. 그래서 codeforge 는 **git = 정본 / Confluence = 읽기 사본** 으로 역할을 못 박고, 흐름을 git→Confluence 한 방향으로만 둔다. 이 안내서는 그 운영·복제·재맞춤 절차를 한곳에 모은 빠른 참조이며, 세부 규범의 단일 원본은 아래 문서들이다.

| 주제 | 단일 원본 |
|---|---|
| 문서 작성·렌더 기준 | [문서 가독성·소통 표준(설계결정 123)](doc-readability-standard.md) |
| 사본 셋업 절차(소비자) | [소비자 적용 안내서 §1o](consumer-guide.md) |
| 공간 구조(체계) 지도 | `docs/confluence-ia-tree.yaml` |
| 사본 진입점 스킬 | `codeforge:confluence-migration` |

## 문서 작성·렌더 기준 (요약)

세부는 [문서 가독성·소통 표준(설계결정 123)](doc-readability-standard.md)이 단일 원본이다. 사본으로 올리는 문서는 git 에서 작성할 때부터 아래를 지킨다.

### 글쓰기 기준

- **제목도 한글로 쓴다.** 경로 접두사와 영어 계약명을 제목에서 뺀다.
- **영어는 도구·제품 고유명사로만 한정한다.** 허용 목록: git, GitHub, Confluence, Jira, Docker, Python, YAML, JSON, API, CI, PR, Claude, Codex, Opus, Sonnet, Haiku, superpowers. 그 밖의 영어는 한글로 옮긴다. (코드·식별자·메타데이터 키처럼 기계가 읽는 토큰은 대상이 아니다.)
- **내부 전문용어는 정해진 한글 대역으로 옮긴다.** 첫 등장 시 괄호로 원어를 병기할 수 있다.
- **격식 문어체**("~한다 / ~이다 / ~된다")로 쓰고, 한자(일·중 포함)를 쓰지 않는다(한자 0).
- 식별자는 한글 라벨에 숫자를 붙여 푼다. 예: "ADR-123" → "설계결정 123".

내부용어 대역 사전(자주 쓰는 항목):

| 원어 | 한글 대역 |
|---|---|
| mirror | 사본 |
| SSOT | 단일 원본 |
| ADR | 설계결정 |
| lane | 작업 단계 |
| wrapper | 본체 |
| story | 작업 단위 |
| drift | 어긋남 |
| gate | 관문 |

### 구조 기준

- 권장 순서를 따른다: **개요 → 배경 → 결정(또는 핵심) → 영향 → (접기) 세부**.
- 핵심을 문서 앞에 둔다(결론 우선). 표·개조식을 우선하고, 긴 평서문 덩어리를 피한다.
- 밀도 높은 거버넌스 세부는 접기로 감춘다.

### 사본 렌더 기준

- 모든 페이지 상단에 **출처 안내 패널**을 둔다. 내용: "원본 = git · 한 방향 사본 · 직접 편집 금지" + 원문 git 경로.
- 묶음·홈·구조(항해) 페이지까지 **한글 제목 + 한글 안내문**을 갖춘다.
- 상태 표시는 한글 뱃지로 한다. 예: `정상` / `주의` / `금지` / `완료`.
- 색 패널·표·접기 같은 시각요소를 적극 활용한다.
- 흐름도(다이어그램)는 외부 이미지·앱 매크로 제약 때문에 **꼭 필요한 소수 페이지에만 수동으로** 넣고, 자동 갱신 대상에서 뺀다.

상단 출처 패널 예시(평문):

```
[note 패널]
원본 = git · 한 방향 사본 · 직접 편집 금지.
출처: <원문 git 경로>
```

상단 출처 패널 예시(HTML 속성):

```html
<div data-type="panel-note">
  원본 = git · 한 방향 사본 · 직접 편집 금지.<br/>
  출처: &lt;원문 git 경로&gt;
</div>
```

## 공간 구조(체계)

사본 공간은 **홈(문서 모음 항해) → 갈래 → 작업 단계별 묶음 → 개별 문서** 순으로 짠다. 갈래·묶음·홈·구조 페이지에도 한글 제목과 한글 안내문, 상단 출처 패널을 둔다.

### 갈래(상위 분류)

사본으로 올리는 문서 종류는 다섯 갈래로 닫혀 있다(임의 확장 금지 — 새 종류는 설계결정 개정이 있어야 함).

| 갈래(한글) | 담는 문서 | 원어 종류 |
|---|---|---|
| 설계 결정 기록 | 어떤 결정을 왜 했는지 적은 기록 | adr |
| 업무 지식 | 도메인·업무 배경 지식 | domain_knowledge |
| 변경 계획 | 무엇을 어떻게 바꿀지 적은 계획 | change_plan |
| 사용·운영 안내서 | 사람이 따라 하는 운영 절차 | orchestrator_playbook |
| 전체 구조 | 살아 있는 아키텍처(구조) 설명 | architecture_doc |

> 위 다섯 갈래에 더해, 부품(plugin)별 묶음과 부품을 가로지르는 공통 묶음(거버넌스 모음 · 구성요소 간 약속 대장 · 본체 구조 등)을 형제 페이지로 둔다. 즉 항해 축은 "부품별 상위 페이지 + 가로지르는 공통 형제 페이지"이다.

### 작업 단계별 묶음

부품(요구사항 / 설계 / 설계리뷰 / 구현 / 구현리뷰 / 통합테스트 / 보안테스트 / 배포 / 배포리뷰 등)마다 묶음 페이지를 두고, 그 아래에 해당 부품에 속한 문서를 모은다. 묶음 페이지에도 한글 제목과 "이 묶음에는 무엇이 들어 있는가"를 안내하는 한글 문장을 넣는다.

### 홈(문서 모음 항해)

공간의 첫 페이지는 전체 문서를 한눈에 둘러보는 항해 페이지로 둔다. 갈래·묶음으로 가는 링크를 한글 제목과 함께 정리하고, 상단에 "이 공간은 git 정본의 읽기 사본"이라는 안내 패널을 둔다.

> 공간 구조의 실제 지도(공간 키·페이지 식별자·부모-자식 관계)는 `docs/confluence-ia-tree.yaml` 이 단일 원본이다.

## 새 소비자에 복제하는 절차

새 소비자(예: mctrader) 공간에 같은 체계를 세우는 4단계이다. 셋업의 단일 원본은 [소비자 적용 안내서 §1o](consumer-guide.md)이며, 아래는 그 요지이다.

### 1단계 — 공간 준비 + 좌표 등록

- 소비자 자기 Confluence 공간을 만든다(권장 — 소유권이 분명하고 부모 페이지 충돌을 피한다). 소규모이거나 공간 공유 계약이 있으면 본체 CFP 공간 아래 하위 묶음을 쓸 수도 있다.
- `.claude/_overlay/project.yaml` 의 `atlassian` 묶음에 좌표를 적는다.

```yaml
# .claude/_overlay/project.yaml
atlassian:
  enabled: true
  confluence:
    base_url: "https://<instance>.atlassian.net"
    space_key: <소비자-공간-키>          # 예: MCT
    homepage_id: "<홈 페이지 식별자>"
    mirror_targets: [adr, architecture_doc]   # 다섯 갈래 중 부분 선택
    api_token_env: CONSUMER_ATLASSIAN_API_TOKEN
    user_email_env: CONSUMER_ATLASSIAN_USER_EMAIL
```

### 2단계 — 갈래·묶음 구조 생성

- `docs/confluence-ia-tree.yaml` 의 소비자 골격(`per_consumer_instantiate_template`)을 복사해 값을 채운다.
- 홈 → 갈래 → 작업 단계별 묶음 페이지를 만든다. 묶음·홈·구조 페이지에도 한글 제목·안내·출처 패널을 넣는다.

### 3단계 — git 문서별 읽기 사본 발행

- git 문서를 위 작성·렌더 기준으로 읽기용 HTML 로 바꿔 발행한다.
- 같은 문서가 이미 페이지로 있으면 그 페이지를 갱신(update)하고, 없으면 새로 만든다(create).
- 처음 발행할 때는 먼저 미리보기(dry-run)로 각 문서가 갱신으로 잡히는지 생성으로 잡히는지 확인한다. 이미 페이지가 있는데 생성으로 잡히면 제목 불일치 신호이므로, 제목을 맞춘 뒤 다시 미리보기를 돌려 갱신으로 바뀐 것을 확인하고 발행한다(중복 페이지 대량 생성 방지).

### 4단계 — 작업 항목 연결

- 각 문서를 관련 작업 항목(GitHub 이슈 또는 Jira 이슈)과 양방향으로 연결한다.
- 이슈 본문·PR 본문에서 설계 내용을 가리킬 때는 git 위치와 Confluence 위치 양쪽 링크를 함께 적는다.

## 변경 시 반영(재이관) 절차

git 문서가 추가·변경되면 다음 순서로 사본을 다시 맞춘다.

| 상황 | 처리 |
|---|---|
| 기존 문서 변경 | 위 작성·렌더 기준으로 다시 렌더 → **같은 Confluence 페이지를 갱신**(제목·번호로 대상 매칭) |
| 새 문서 추가 | 해당 갈래·묶음 아래에 **새 페이지 생성** |
| 문서 폐기 | 사본 페이지에 폐기 안내를 표시(직접 삭제는 신중히 — 정본 상태에 맞춤) |

핵심 주의:

- **대상 매칭은 제목·번호로 한다.** 제목이 어긋나면 갱신 대신 새(중복) 페이지가 만들어져 공간이 오염된다. 발행 전 미리보기로 갱신 여부를 확인한다.
- **한 번에 하나씩 처리한다.** 여러 페이지를 한꺼번에 몰아치면 요청 제한(429)에 걸리기 쉽다. 순차로 처리해 제한을 피한다.

## 자동화 로드맵 (A-프라임 운영 모델)

운영 모델은 **A-프라임**이다 — git 은 글을 쓰는 정본(원천)으로 그대로 두고, Confluence 를 단순 "읽기 사본"에서 **공식 "정본 읽기 + 사람 편집 화면"**으로 끌어올린다. 사람은 Confluence 에서 읽고 편집할 수 있으며, 그 편집은 git 으로 되돌아간다(제안). 이는 git=정본 결정을 뒤집는 것이 아니라 **넓히는 것**이다 — git 이 여전히 최종 정본이고, 되돌리기 경로는 손실을 허용하는 "제안"일 뿐이다.

| 채널 | 역할 |
|---|---|
| git | 글을 쓰는 정본(원천) — 에이전트가 글을 쓰고, 검토·검사·버전 관문을 거친다. 변하지 않음. |
| Confluence | 공식 "정본 읽기 + 사람 편집 화면" — 사람이 읽고 편집할 수 있다(끌어올림). |

> 핵심: Confluence 가 사람 편집 화면으로 올라가더라도 **글을 쓰는 정본은 여전히 git** 이다. Confluence 편집은 git 변경 제안으로 되돌아가고, 사람이 그 제안을 검토·반영해야 정본에 들어간다(아래 §"되돌리기" 참조).

### 앞 방향 — git → Confluence 자동 (forward)

git 문서가 바뀌어 본 가지(main)에 반영되면 사본을 자동으로 다시 맞춘다. 이는 위 한 방향 흐름의 자동화이다(새 흐름이 아니다).

핵심: **git 문서를 고치는 주체는 언제나 Claude 세션**(codeforge 에이전트 또는 총괄 세션)이다. 그러므로 앞 방향 동기화의 **1차(기본) 경로는, 그 세션이 같은 흐름에서 Atlassian 연결(MCP)로 Confluence 페이지를 직접 갱신**하는 것이다 — CI·비밀값(secret)·예약 실행(cron)이 필요 없다. 이는 처음 250여 페이지를 손으로 이관한 그 방식과 같고, 앞 방향은 그것을 변경분에만 적용한다.

#### 1차 경로 — Claude 세션이 연결(MCP)로 직접 갱신 (비밀값 0)

| 단계 | 내용 |
|---|---|
| 시점 | **문서 PR 머지 후처리**(설계결정 026, 머지 후 정리 단계) — 머지를 확인한 세션이 이어서 수행한다. |
| 변경분 추리기 | 머지된 PR 의 변경 문서 중 사본 대상(아래 다섯 갈래 + 안내서)만 고른다. 진행 산출물(작업 단위·고침 기록·근거 묶음 등)은 뺀다. |
| 재작성 | 세션(Claude)이 읽기 표준(설계결정 123)으로 읽기용 HTML 로 다시 쓴다(별도 AI 키 불요 — 세션 자체가 수행). |
| 발행 | 같은 흐름에서 Atlassian 연결(MCP)로 해당 페이지를 갱신한다(좌표는 `docs/confluence-mirror-manifest.yaml`, 없으면 새로 만들거나 "처음은 사람이 수동 발행"으로 기록). |

- **비밀값(secret) 0**: 세션은 자기 연결 권한(OAuth)으로 Confluence 를 갱신하므로 Confluence 자격증명·AI 키를 따로 주입할 필요가 없다.
- **세션 한정**: 이 연결 권한은 총괄 세션 안에서만 유효하다. 하위 작업(subagent)으로 위임하면 인증이 끊기므로(401), 연결 갱신은 총괄 세션이 직접(inline) 수행한다.

#### 보조(선택) 경로 — 무인(headless) CI 자동 발행

세션 없이 무인으로 돌려야 할 때(예: 사람·에이전트 개입 없는 야간 배치)에만 쓰는 **선택적 보조 경로**이다. CI 실행기에는 세션 연결이 없으므로, 이때만 비밀값(토큰)이 필요하다.

| 단계 | 내용 | 필요한 것 |
|---|---|---|
| 변경 감지 | 본 가지(main)에 글이 바뀌면 CI 가 알아챈다 | CI 트리거 (`.github/workflows/confluence-forward-sync.yml`) |
| 자동 재작성 | CI 안에서 Claude 가 읽기 표준(설계결정 123)으로 읽기용 HTML 로 다시 쓴다 | AI(LLM) API 키 (`ANTHROPIC_API_KEY`) |
| 자동 발행 | 결과를 같은 Confluence 페이지에 올린다(좌표는 `docs/confluence-mirror-manifest.yaml`) | Confluence 자격증명 비밀값(secret) |

- 보조 경로 발행 스크립트 = `scripts/confluence_forward_sync.py` (모드 둘 — `--build-manifest` 좌표표 채우기 / 기본 발행 모드). **이 스크립트·CI 워크플로는 1차 경로가 아니라 무인 보조(fallback)이다.**
- **비밀값이 없으면 미리보기(dry-run)**: AI·Confluence 비밀값이 없으면 실제 호출을 하지 않고(0건), 무엇이 바뀌었고 어느 페이지로 갈지만 기록한다(안전 동작).
- 좌표표(`docs/confluence-mirror-manifest.yaml`)에 없는 문서는 "처음은 사람이 수동 발행 필요"로 기록하고 건너뛴다(자동 신규 생성은 중복 페이지 위험으로 막는다).

### 되돌리기 — Confluence 편집 → git 변경 제안 (backward)

사람이 Confluence 페이지를 직접 고친 경우, 그 편집을 git 으로 되돌린다 — **단 "제안"(변경 제안) 수준이며, git 이 최종 정본**이다. (이 경로는 설계만 확정했고 실제 구축은 후속 과제이다 — 아래 §"되돌리기 설계" 참조.)

되돌리기도 1차 경로는 앞 방향과 같은 원리이다 — **Claude 세션이 Atlassian 연결(MCP)로 바뀐 페이지를 읽어** 되변환한 뒤 git 변경 제안(PR)을 만든다(비밀값·예약 실행 불요). 무인 예약 점검(cron)은 선택적 보조 경로이다.

> 위 비밀값(API 키·Confluence 자격증명)은 **무인 보조(CI) 경로에만** 필요하다. 1차 경로(세션이 연결로 직접 갱신)는 비밀값 없이 동작한다. 따라서 앞 방향 사본 맞춤은 세션이 문서를 머지하는 정상 흐름에서 비밀값 없이 즉시 동작하며, 비밀값은 세션 없이 무인으로 돌릴 때만 필요하다. 되돌리기는 현재 설계 단계이다.

## 되돌리기 설계 — Confluence 편집 → git 변경 제안 (backward, 설계만)

사람이 Confluence 에서 문서를 고치면 그 편집을 git 으로 흘려보내되, **자동으로 정본을 덮어쓰지 않는다** — 변경 제안(PR)으로만 올린다. git 이 최종 정본이고, 사람이 그 제안을 검토·반영해야 정본에 들어간다.

| 단계 | 내용 |
|---|---|
| 변경 감지 | 주기적 점검(예: 정해진 시간마다)으로, 페이지의 판(version)이 마지막 발행 표식(sync 표식) 이후 올라간 페이지를 찾는다. |
| 되변환 | Confluence 저장 형식(ADF) → markdown 으로 되돌린다(**손실 허용** — 왕복 변환이 정확하지 않으므로 "초안 제안" 수준). |
| 변경 제안 | 되돌린 markdown 을 가지(branch)에 담고 `gh pr create` 로 **변경 제안(PR)** 을 만든다. 사람이 검토·정정 후 반영한다. |

설계 원칙:

- **손실 허용(제안 수준)**: ADF → markdown 왕복은 정확하지 않다. 변환 결과는 "초안 제안"일 뿐이고, 사람 검토가 정확성 관문이다. git 이 글을 쓰는 정본이므로 변환이 어긋나도 사람이 반영하기 전까지 정본은 그대로다.
- **직접 덮어쓰기 아님**: 되돌리기는 Confluence → git **직접 쓰기가 아니다** — 변경 제안(PR)이다. 따라서 한 방향(git → Confluence) 흐름 원칙을 깨지 않는다(Confluence → 본체로 들어오는 자동 쓰기 통로 0건).
- **구현은 후속**: 점검·되변환·제안 생성의 실제 구축은 복잡하고 비밀값이 필요하므로 후속 과제이다(본 절은 설계만). 그 전까지 사람이 Confluence 편집을 git 에 손으로 반영한다.

## 운영 노하우 (제약)

Confluence 의 사본 렌더는 다음 제약을 지켜야 깨지지 않는다. 위반하면 페이지가 비거나 깨진 모양으로 발행된다.

| 항목 | 제약 |
|---|---|
| 패널 형식 | 패널은 HTML 속성 `data-type="panel-note"` / `panel-info` / `panel-warning` / `panel-success` 형식만 쓴다. |
| 패널 안 표 | 패널 안에 표를 넣지 않는다. |
| 표 구조 | 표에서 칸 병합(colspan)·빈 칸·머리행 분리를 쓰지 않는다. |
| 금지 태그 | `time` 태그·`hr`(가로줄) 태그를 쓰지 않는다. |
| 본문 기호 | 본문의 꺾쇠(`<` `>`)와 앰퍼샌드(`&`)는 이스케이프한다(`&lt;` `&gt;` `&amp;`). |
| 외부 이미지 | 외부 이미지는 차단된다. 앱(매크로) 없이 다이어그램을 자동으로 그릴 수 없다. |

> **다이어그램 처리**: 외부 이미지 차단과 앱 매크로 제약 때문에 다이어그램을 자동으로 렌더할 수 없다. 따라서 다이어그램은 자동 갱신 대상에서 빼고, 꼭 필요한 소수 페이지에만 수동으로 넣는다.

### 사본 금지 다섯 영역

다음 다섯 영역은 사본으로 절대 만들지 않는다(진행 산출물 — git·이슈에만 둔다).

1. 작업 단위 파일(`docs/stories/<KEY>.md`)
2. 고침 기록(작업 단위 파일 §10 하위)
3. 근거 묶음(작업 단위 파일 §14 하위)
4. 결정 묶음(`decisions/<식별자>.yaml`)
5. 생성 지시문(한 번 쓰고 사라지는 임시 지시)
