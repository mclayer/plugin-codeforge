# codeforge 이해 문서 (UNDERSTANDING)

> 작성 방법: 대량 에이전트 검토 2회가 환경 문제(구조화출력 실패 / 서버 요청제한)로 빈손으로 끝남.
> 대신 **결정 기록 119개 머리말 일괄 추출 + 검사 등록부(evidence-checks-registry.yaml) 집계**로 구조를 직접 읽음.
> 이 문서의 모든 수치는 실측이며, 추측이 아님. 다만 "각 항목 본문 정독에 의한 항목별 가치 판정"은 미완(아래 §7).

---

## 1. 이 시스템이 무엇이고 왜 이렇게 자랐나

codeforge는 Claude Code로 소프트웨어 개발 전 과정(요구사항→설계→리뷰→구현→배포)을 자동 진행하려는 오케스트레이션 플러그인이다. 정작 이 저장소엔 **제품 코드가 0줄**이다 — 전부 "어떻게 일할지를 규정하는 규칙"과 "그 규칙을 감시하는 기계"다.

자란 방식이 핵심이다. **사고가 터질 때마다 규칙을 하나씩 더했다.** 결정 기록(ADR) 119개, 검사 스크립트 146개, CI 워크플로 145개가 그 누적이다. 문제는 **더하기만 하고 빼지 않았다는 것** — 시스템 자체 회고에도 "폐기 0건 / 규칙이 쌓이기만 함"이 적혀 있다.

결정 기록을 분류해 보면 의도가 드러난다. 119개 중 **73개(61%)가 "거버넌스/팀·프로세스"** — 즉 제품이 아니라 *일하는 방식을 통제하는 규칙*이다. 실제 시스템 설계(아키텍처·도구·에이전트설계)에 관한 건 ~20개뿐이다. **시스템이 자기 자신을 통제하는 데 대부분의 노력을 썼다.**

검사 기계는 더 극적이다. 등록된 검사 146개 중 **140개(96%)가 "경고(warning)" 등급 — 아무것도 막지 않는다.** 실제로 병합을 막는 건 10개뿐. 나머지 140개는 CI 시간과 유지보수 비용만 쓰면서 무시되는 경고를 뿜는다. **"강제"라는 거대한 외형이 실제로는 거의 강제하지 않는다.**

그 결과가 "나쁜 결과물"이다. 매 턴 모델은 2만7천 토큰짜리 지침을 깔고 시작하고, 96%가 헛도는 검사망과 자기들끼리 모순된 문서 사이에서 일한다. 절차의 외형은 완벽한데 산출물은 평범한, 전형적인 과(過)거버넌스 상태다.

## 2. 집계 (실측)

### 결정 기록(ADR) 119개 — 분류
| 분류 | 개수 | 성격 |
|---|---:|---|
| governance | 46 | 메타 (규칙의 규칙) |
| team & process | 27 | 메타 |
| tooling-infrastructure | 8 | 일부 실질 |
| architecture | 8 | **실질 (제품 설계)** |
| orchestration(+discipline) | 10 | 혼합 |
| workflow-policy | 5 | 메타 |
| agent-design | 4 | **실질** |
| security | 2 | **실질** |
| 기타(lifecycle/process/audit 등) | 9 | 혼합 |

→ **메타 성격(거버넌스+팀프로세스+워크플로정책+감사) ≈ 73개(61%)** vs 제품 실질(아키텍처+에이전트설계+보안+도구) ≈ 22개.

### 검사 146개 — 등급 (이게 가장 중요한 수치)
| 등급 | 개수 | 의미 |
|---|---:|---|
| warning (비차단) | **140 (96%)** | 막지 않음 — 무시되는 경고 + 비용만 발생 |
| blocking (차단) | **10 (4%)** | 실제로 병합을 막음 = 진짜 일하는 것 |

### 상태(status) 어휘 — 유지 안 됨의 증거
| 표기 | 개수 |
|---|---:|
| Accepted | 87 |
| Proposed | 15 |
| Active | 9 |
| Adopted | 5 |
| Superseded/Deprecated | 3 |

→ **같은 "살아있음"을 Accepted/Active/Adopted 세 단어로 표기** (표준 어휘 없음). "Proposed"인데 이미 차단형 CI로 강제되는 것 다수(worktree 규칙, 회고 의무 등) — **상태 필드가 현실과 따로 논다.**

## 3. 진짜 일하는 핵심 (유지)

| 영역 | 근거 | 없으면 |
|---|---|---|
| 차단형 검사 10개 | registry에서 blocking 등급 | 깨진 코드/문서가 main에 병합됨 |
| main 보호 6종 필수 체크 | branch protection | 게이트 무력화 |
| 스킬 12개 | on-demand 로드(매 턴 아님) | 오케스트레이션 노하우 소실 — **비대화 안 일으킴, 좋은 패턴** |
| 아키텍처·에이전트설계·보안 ADR ~22개 | category 실측 | 실제 설계 결정 근거 소실 |
| 핵심 템플릿(story/PR/ADR/change-plan) | 산출물 양식 | 에이전트 출력 일관성 |
| 세션 초기화 hook | session-start | 플러그인 부팅 |

## 6. 근거 기반 권고 — "70% 삭제" 추정 교정

이전 PRUNE-PLAN.md의 "70% 삭제"는 개수 기반 추측이었다. **이번 구조 데이터로 검증한 결과: 방향은 맞고, 오히려 보수적이었다.**

| 대상 | 데이터 근거 | 권고 |
|---|---|---|
| 경고검사 140 | 96%가 비차단 | **대부분 삭제/통합** — 차단 가치 있는 것만 차단형으로 승격, 나머지 제거 |
| 거버넌스 ADR ~73 | 61%가 메타 | **압축 보관** — 1장짜리 "현행 결정 + 배운 교훈" 로 요약, 원본은 archive |
| 아키텍처/설계 ADR ~22 | 제품 실질 | **유지** |
| 죽은 ADR 5~6 | 명시적 폐기 | **즉시 archive** |
| 번호 충돌 4건 | 뼈대 손상 | **재번호 또는 통합** |
| 매 턴 hook 대화규칙 | 토큰 비용 | **제거** |
| CLAUDE.md 27K토큰 | 매 턴 로드 | **≤2.5K 재작성** |

**한 줄 결론: 삭제 규모를 "개수 70%"가 아니라 "비차단 검사 96% + 메타 ADR 61%를 압축"으로 재정의.** 차단형 10개 + 실질 ADR 22개 + 스킬 12개가 진짜 골격이다.

## 7. 불명 · 사람이 직접 봐야 할 항목

- **항목별 정밀 가치 판정 미완**: 대량 검토 2회 실패로, "이 경고검사 140개 중 *정확히* 어느 것이 그래도 쓸모 있나"는 아직 한 건씩 안 봤다. 삭제 실행 전 차단형 10개 + 후보 경고검사는 한 번씩 확인 필요.
- **계약 구버전 안전 삭제 여부**: review-verdict v1~v3 등 구버전을 consumer(mctrader 등)가 아직 참조하는지 확인 후 삭제.
- **ADR-076/083 처리**: "대체됨"인데 Active — 실제로 죽었는지 살았는지 사람이 판단.
- **번호 충돌 4건**: 재번호 시 외부 참조(다른 문서/PR 인용) 깨짐 주의.
