# codeforge 슬림화 효과 테스트 (A/B)

목적: CLAUDE.md 96% 슬림 + 대화규칙 hook 제거가 **실제 결과물 품질**을 높이는지 검증.

## 실행 방법
1. **B (슬림판)**: 현재 브랜치 `prune/cut-now-unwired` 체크아웃 상태에서 **새 세션** 시작 → 아래 프롬프트 붙여넣기.
2. **A (원본)**: `git checkout main` 후 **새 세션** 시작 → 같은 프롬프트 붙여넣기.
3. 두 결과를 아래 채점표로 비교.

> 두 세션 모두 Orchestrator 모델 = opus 로 동일하게. 작업은 같은 프롬프트 한 글자도 안 바꾸고.

---

## 붙여넣을 테스트 프롬프트 (A/B 동일)

```
codeforge로 작은 작업 하나 해줘.

consumer 프로젝트의 .claude/_overlay/project.yaml 이 필수 키를 갖췄는지 검사하는
작은 파이썬 CLI 스크립트를 만들어줘. 요구사항:

- 입력: project.yaml 경로 (인자 1개, 기본값 .claude/_overlay/project.yaml)
- 검사: 다음 최상위 키가 모두 존재하는지 — github_org, story_key_prefix, codeowners_team
- 누락 키가 있으면 누락 목록을 stderr에 출력하고 exit code 1
- 모두 있으면 "OK" 출력하고 exit code 0
- yaml 파싱 실패 시 명확한 에러 메시지 + exit code 2

테스트도 함께. 끝나면 스크립트 경로와 테스트 통과 결과를 보여줘.
```

## 채점표 (A vs B 비교)

| 항목 | 무엇을 보나 | A(원본) | B(슬림) |
|---|---|---|---|
| 1. 결과물 정확성 | 스크립트가 요구사항 4개(키검사/누락 exit1/OK exit0/파싱실패 exit2)를 실제로 충족하나 | | |
| 2. 테스트 실제 통과 | "통과"라고만 말하지 않고 실제 실행 결과를 보여줬나 | | |
| 3. 걸린 턴 수 | 결과까지 사용자 개입 몇 번 | | |
| 4. 의식(ceremony) 비중 | Story/ADR/라벨/검사 절차에 얼마나 시간을 썼나 (작업 대비) | | |
| 5. 응답 명료성 | 내부 용어 범벅인가, 핵심이 한눈에 보이나 | | |
| 6. 삼천포 여부 | 거버넌스 점검하다 본 작업을 놓치거나 헤맸나 | | |

## 판정 기준
- **B가 1·3·4·6에서 명확히 낫다** → 슬림화 성공, 병합 + 나머지 정리 진행.
- **차이 미미** → 슬림화는 무해하나 결과 품질 주범은 따로 (검사망·서브에이전트 환경 등) → 그쪽 조사.
- **B가 나쁘다** → 빠진 핵심 규칙 있음 → CLAUDE.md 초안 보강 후 재테스트.

## 주의 (혼동 방지)
- 이번(현재) 세션은 hook이 이미 로드돼 대화규칙이 계속 찍힌다 — **반드시 새 세션**에서 테스트.
- B 세션에서 "필수 플러그인/검사 누락" 류 경고가 줄어든 것은 정상(의도된 제거).
